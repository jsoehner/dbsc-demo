# Architecture & Data Flow

This document outlines the flows demonstrated in the DBSC demo application.

## 1. Authentication Flow (DBSC Enabled)

When DBSC is enabled, the login process requires the client device to generate a cryptographic key pair and register the public key with the server.

```mermaid
sequenceDiagram
    participant U as User
    participant B as Browser (Victim)
    participant S as Server
    
    U->>B: Enters Credentials & Enables DBSC
    B->>B: Generate Non-exportable KeyPair (WebCrypto)
    B->>S: POST /login {username, password, public_key}
    S->>S: Validate Credentials
    S->>S: Create Session & Bind public_key to Session ID
    S-->>B: Return HTTP-Only Cookie (session_id)
```

## 2. Authenticated Data Request (DBSC Enabled)

Every subsequent request must include a cryptographic signature generated by the private key, proving the request originates from the bound device.

```mermaid
sequenceDiagram
    participant B as Browser (Victim)
    participant S as Server
    
    B->>B: Create payload (timestamp)
    B->>B: Sign payload using Private Key
    B->>S: GET /data <br/> Header: Cookie: session_id<br/>Header: X-Timestamp<br/>Header: X-Signature
    
    S->>S: Lookup Session by Cookie
    S->>S: Extract bound public_key
    S->>S: Verify Timestamp is recent
    S->>S: Verify X-Signature using public_key
    
    alt Verification Successful
        S-->>B: Return 200 OK (Sensitive Data)
    else Verification Failed
        S-->>B: Return 403 Forbidden
    end
```

## 3. Cookie Theft & Replay Attack (Standard Cookie)

If DBSC is NOT enabled, the server only requires the cookie. If the cookie is stolen, the attacker has full access.

```mermaid
sequenceDiagram
    participant B as Browser (Victim)
    participant M as Malware
    participant H as Hacker
    participant S as Server
    
    Note over B, M: Victim is logged in (Standard Session)
    M->>B: Extract Session Cookie
    M->>H: Exfiltrate Cookie
    
    H->>S: POST /hacker_access <br/> Header: Cookie: stolen_session_id
    S->>S: Lookup Session by Cookie
    S->>S: Validate Session (Success)
    S-->>H: Return 200 OK (Sensitive Data Leaked!)
```

## 4. Failed Replay Attack (DBSC Enabled)

When DBSC is enabled, stealing the cookie is insufficient. The attacker cannot steal the hardware-bound private key.

```mermaid
sequenceDiagram
    participant B as Browser (Victim)
    participant M as Malware
    participant H as Hacker
    participant S as Server
    
    Note over B, M: Victim is logged in (DBSC Session)
    M->>B: Extract Session Cookie
    M->>H: Exfiltrate Cookie
    Note over M, H: Malware cannot extract the Private Key!
    
    H->>H: Attempt to forge signature
    H->>S: POST /hacker_access <br/> Header: Cookie: stolen_session_id <br/> Header: X-Signature: fake_sig
    
    S->>S: Lookup Session by Cookie
    S->>S: Extract bound public_key
    S->>S: Verify X-Signature using public_key
    Note over S: Verification FAILS (Invalid Signature)
    S-->>H: Return 403 Forbidden (Attack Blocked)
```
